# Phase 01 — UI Design Contract

> Visual and interaction contract for frontend phases. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

---

## Design System

| Property | Value |
|----------|-------|
| Tool | none |
| Preset | not applicable |
| Component library | none; Laravel Blade components/partials only |
| Icon library | inline SVG only; no package required |
| Font | system UI stack: `ui-sans-serif, system-ui, -apple-system, BlinkMacSystemFont, "Segoe UI", sans-serif` |

**Implementation contract:** use Blade + plain CSS/Vite assets. Do not require React, Vue, Inertia, Livewire, Tailwind, or shadcn for this MVP.

---

## Spacing Scale

Declared values (must be multiples of 4):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon gaps, inline padding |
| sm | 8px | Compact element spacing |
| md | 16px | Default element spacing, form field gaps |
| lg | 24px | Card padding, mobile section padding |
| xl | 32px | Desktop layout gaps, form groups |
| 2xl | 48px | Major section breaks |
| 3xl | 64px | Page-level spacing on desktop |

Exceptions: interactive controls must be at least 44px high on mobile; QRIS image container may use fixed 280px mobile / 360px desktop max width.

---

## Typography

Use exactly these sizes and only weights 400 and 600.

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Body | 16px | 400 | 1.5 |
| Label | 14px | 600 | 1.4 |
| Heading | 24px | 600 | 1.2 |
| Display | 40px | 600 | 1.1 |

**Responsive behavior:** on narrow screens keep the same tokens, but constrain display text width to avoid more than 4 lines in the hero.

---

## Color

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `#FFF7ED` | Page background, warm sales surface |
| Secondary (30%) | `#FFFFFF` | Cards, forms, invoice panels, admin table rows |
| Accent (10%) | `#F97316` | Primary CTA, price anchor, countdown numbers, active download button |
| Destructive | `#DC2626` | Reject payment action and rejected-state alert only |

Accent reserved for: primary “Dapatkan Sekarang” CTA, checkout submit button, verified download button, soft CTA on lead magnet page, price `Rp 99.000`, countdown numeric cells, and focus rings.

State indicators: pending/proof-submitted use amber text/border `#D97706`; verified uses green text/border `#16A34A`; rejected uses destructive. These colors are limited to badges, alerts, and status labels, not general branding.

Text colors: primary `#111827`, secondary `#4B5563`, muted `#6B7280`, border `#FED7AA` on warm sections and `#E5E7EB` on admin surfaces.

## Page and Interaction Contracts

### 1. Landing Page `/`

Required section order:
1. Hero with PAS headline, short benefit subcopy, CTA “Dapatkan Sekarang”, price Rp 99.000 near CTA, and secondary lead magnet CTA.
2. Problem/agitation section for time wasted creating prompts, inconsistent output, and content bottlenecks.
3. Solution/benefits section for faster marketing, content, and business workflows.
4. Product preview showing ebook/ZIP contents and prompt swipe-file examples; use real product content snippets only.
5. Pricing anchor card with Rp 99.000, bundle contents, guarantee/trust badges, and CTA.
6. Configurable countdown promo; countdown text must say promo deadline, not fake stock scarcity.
7. Lead magnet block: email-only form for “50 Prompt Pemasaran Gratis”.
8. FAQ / support note explaining QRIS manual and post-verification download.

Trust badges allowed: “Pembayaran QRIS manual”, “Download setelah verifikasi”, “File ZIP bundle”, “Tanpa login pembeli”, “Email-only lead magnet”.

### 2. Checkout / Order Creation

Order form fields: name, email, WhatsApp. Use one column on mobile and two-column layout only on desktop if it does not reduce readability.

Button copy: “Buat Order & Lihat QRIS”. After submit, redirect to invoice token page. Never require buyer account creation.

### 3. Invoice / QRIS Page `/orders/{token}`

Pending state must show:
- status badge “Menunggu Pembayaran”;
- fixed amount Rp 99.000;
- QRIS image area;
- steps: scan QRIS, bayar sesuai nominal, simpan bukti, upload bukti;
- proof upload form with JPG/PNG/WebP note.

Proof submitted state must show:
- status badge “Bukti Terkirim”;
- message “Admin akan mengecek pembayaran secara manual.”;
- keep re-upload affordance visible but secondary.

Rejected state must show:
- destructive alert with reject reason;
- primary re-upload button “Upload Ulang Bukti”.

Verified state must show:
- verified badge;
- primary button “Download ZIP Ebook”;
- support note telling buyer to save the invoice link because it remains active.

### 4. Admin Verification UI

Admin dashboard layout: simple authenticated Blade page, desktop table + mobile stacked cards.

Order list priority: `proof_submitted` first, then `pending`, `rejected`, `verified`. Each row/card must show buyer name, email, WhatsApp, amount, status, upload time, proof preview link, Confirm, and Reject.

Confirm action copy: “Confirm”. Reject action copy: “Reject”. Reject opens or reveals a required short reason field before submit. No delete action in MVP.

### 5. Lead Magnet Flow

Landing opt-in field: email only. Submit button: “Kirim & Download Gratis”. Duplicate emails still continue to the download page.

Download page must show:
- heading “Download 50 Prompt Pemasaran Gratis”;
- primary button “Download Lead Magnet”;
- soft CTA card: “Butuh sistem prompt yang lebih lengkap?” with button “Lihat Ebook Rp 99.000”.

## Desktop and Mobile Behavior

| Area | Mobile | Desktop |
|------|--------|---------|
| Landing hero | Single column, CTA visible without horizontal scroll | Two-column hero allowed: copy left, product/price card right |
| CTA | Full-width buttons, 44px min height | Buttons can be inline, 44px min height |
| Pricing card | Sticky not required; place after benefits | Prominent card near hero and repeated before FAQ |
| QRIS page | QRIS centered, instructions below | QRIS and instructions may sit side-by-side |
| Admin orders | Stacked cards with action buttons | Table/list with right-aligned actions |
| Upload forms | Full-width file input and submit button | Card layout with clear file constraints |

No interaction may depend on hover only; every action must work with touch and keyboard.

---

## Accessibility and Form States

- Every input requires visible label and server-side validation error area.
- Focus ring must use accent `#F97316` with at least 2px outline/box-shadow.
- Buttons must have loading/disabled state copy: “Memproses...” for form submits.
- File upload helper text: “Format: JPG, PNG, atau WebP. Maksimal 4 MB.”
- Countdown must include readable text fallback; do not communicate urgency by color alone.
- Status badges must include text labels, not color-only indicators.
